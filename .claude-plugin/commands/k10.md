# K10 — UniHiker K10 Hardware Reference

The UniHiker K10 is an ESP32-S3 with built-in USB-Serial-JTAG (`0x303a:0x1001`),
240x320 color display, OV2640 camera, capacitive touchscreen, and WiFi. One CDC
port appears as `/dev/cu.usbmodem*` (macOS) or `/dev/ttyACM0` (Linux).

Firmware: `k10_micropython_v0.9.8.bin` (MicroPython 1.26, ~16.5MB). Recovery
image lives at `~/Downloads/k10_micropython_v0.9.8.bin`.

Project repo: `~/WORKSPACE/k10-polaroid-booth`

## Hardware API (verified on real hardware)

### Camera

```python
from k10_base import Camera
cam = Camera()
cam.init()                      # emits "cam_init ok"; call once
raw = cam.camera_capture()      # 153600 bytes raw RGB565 (240x320, BGR;16 LE)
cam.save(path)                  # save to filesystem
```

153600 = 240 * 320 * 2 bytes/pixel. Server decodes as:
`Image.frombuffer("RGB", (240,320), raw, "raw", "BGR;16", 0, 1)`

### Screen

```python
from unihiker_k10 import screen   # THE singleton — only valid import
screen.init(dir=2)                # portrait 240x320
screen.show_camera(cam)           # start live viewfinder (DMA-safe)
screen.stop_camera()              # stop viewfinder
screen.show_bg(color=0x0C0E1C)    # fill background
screen.draw_rect(x=, y=, w=, h=, bcolor=, fcolor=)
screen.draw_text(text="ASCII", x=, y=, font_size=, color=)
screen.show_draw()                # flush to panel
screen.clear()
```

Key methods: `init`, `clear`, `deinit`, `show_bg`, `show_draw`, `show_camera`,
`stop_camera`, `draw_rect`, `draw_text`, `draw_circle`, `draw_line`, `draw_point`.

### Touch (FT6336)

```python
import k10_base
_i2c = k10_base.k10_i2c          # I2C0, scl=48 sda=47
_TOUCH_ADDR = 0x38

def touched() -> bool:
    return (_i2c.readfrom_mem(0x38, 0x02, 1)[0] & 0x0F) > 0
```

Works with camera active. Independent of GPIO5/GPIO11 (camera bus).

### WiFi

```python
from k10_base import WiFi
wifi = WiFi()
wifi.connect(ssid="name", psd="pass", timeout=30000)  # BLOCKING
wifi.status()                                          # truthy when connected
```

MUST connect BEFORE `screen.init()` — display framebuffer starves WiFi DMA.

### Boot Guard (required in all firmware)

```python
import sys, select
_p = select.poll()
_p.register(sys.stdin, select.POLLIN)
if _p.poll(3000):
    sys.exit()   # abort to REPL on any keypress
```

`utime.sleep(3)` does NOT work as a guard (never reads stdin on USB-Serial-JTAG).

## Hard-Fault Traps

| Don't | Result | Do Instead |
|-------|--------|-----------|
| `import unihiker_k10` (bare) | InstructionFetchError, reboot | `from unihiker_k10 import screen` |
| `k10_base.Screen()` (instantiate) | LoadProhibited, reboot | use `screen` singleton |
| `screen.show_camera_feed(cam)` | AttributeError: 'Screen' has no 'desc' | use `screen.show_camera(cam)` |
| Soft reset (Ctrl+D) to re-run firmware | re-init faults | hard reset via esptool |
| Manual draws while camera + WiFi active | HANG (GDMA exhaustion) | headless mode |
| Camera + WiFi + Screen simultaneously | ENOMEM (I2S can't allocate DMA) | pick 2 of 3 |
| asyncio with camera | event loop conflicts | plain sync loop |

## GDMA Channel Constraint (the fundamental limitation)

ESP32-S3 has 5 GDMA channels. On cold boot:
- Camera parallel bus: consumes channels (must be first)
- WiFi SPI DMA: consumes channels (must be second)
- Screen I2S bus: CANNOT allocate — all channels taken

**You can have camera + WiFi OR camera + screen, but NOT all three.**

This is NOT a software bug — it's a hardware resource limit. `wlan.active(False)` and
`screen.deinit()` do NOT release GDMA channels (ESP-IDF driver limitation in MicroPython).

### What works on cold boot
- Camera + WiFi + Touch (headless): capture + upload, no display
- Camera + Screen (no WiFi): viewfinder + local save, no upload
- Camera alone: REPL testing

### Why REPL sessions appeared to work with all three
The REPL inherits DMA state from whatever the crashed/interrupted `main.py` already
allocated. If camera was initialized first (from a prior boot), and screen was imported
(from the same or prior boot), both have their channels from separate init sequences.
This does NOT reproduce on a clean cold boot from `main.py`.

## Correct Boot Order (production)

1. Camera init (`Camera().init()`) — claims GDMA channels for parallel capture
2. WiFi connect (blocking) — claims remaining GDMA channels for SPI
3. Touch-poll loop (I2C at 0x38 — no DMA, always works)
4. On touch: `camera_capture()` → POST to server → gallery shows polaroid live

## Key Findings (2026-06-02)

- `show_camera(cam)` works (no DMA hang) **only without WiFi**
- `show_camera_feed(cam)` is BROKEN (firmware bug: looks for `self.desc`, attribute is `self.img_dsc`)
- `camera_capture()` returns 153600 bytes in all states (with/without feed, with/without WiFi)
- Speaker also uses I2S → same ENOMEM constraint as screen
- A/B face buttons (GPIO5/GPIO11) UNUSABLE with camera initialized (shared parallel bus)
- Touch detection via I2C is fully independent and works in all states
- WiFi disconnect (`wlan.active(False)`) does NOT free GDMA channels for screen
